  📘 第一部分：核心概念教学

  在开始写计划之前，我们先搞清楚两个最基础的词：DevOps 和 Cloud Engineering。

  1. 什么是 DevOps？
  很多人以为 DevOps 就是 Jenkins + Docker + K8s，其实它更像是一种文化和流程的“翻译官”。
   * Dev (Development)：想要快速发布新功能。
   * Ops (Operations)：想要系统稳如老狗，不挂掉。
   * DevOps：打破两者的隔阂，通过自动化（Automation）让开发和运维同步，实现“高质量的快速交付”。

  2. 什么是 Cloud Engineering（云工程）？
  如果说 DevOps 是流程，那 Cloud 就是战场。
  过去我们要买物理服务器、拉网线（On-Premise）；现在我们通过代码直接在 AWS/Azure
  上“变出”成千上万台服务器。这就是 Infrastructure as Code (IaC，基础设施即代码)。

  📘 第一部分：核心教学 —— Linux 文件系统与存储自动化

  在 Day 01 中，你提到过修改 fstab 来挂载 Windows
  游戏硬盘。那是手工时代的巅峰，但现在我们要进入自动化时代。

  1. 什么是 `fstab`？
  fstab (File Systems Table) 是 Linux 内核启动时读取的配置文件。它决定了哪些分区挂载到哪个目录下。
   * 痛点：如果你手动改错了 fstab，Arch 可能会进入紧急模式（Emergency Mode），让你进不去系统。
   * DevOps 方案：使用 UUID（Universally Unique Identifier）而不是 /dev/sdb1 这种会变动的名称。

  2. 新工具教导：`lsblk` 与 `blkid`
   * `lsblk`: 以树状列出所有块设备（硬盘、分区）。
       * 用法：lsblk -f (列出文件系统类型和 UUID)。
   * `blkid`: 查看更详细的块设备属性，通常用于获取精准的 UUID。

  📘 第一部分：纠偏与进阶 —— 权限的“数学逻辑”

  1. 权限数值纠偏 (非常重要！)
  在 Linux 中，权限是基于二进制的：
   * Read (r) = 2^2 = 4
   * Write (w) = 2^1 = 2
   * Execute (x) = 2^0 = 1

  所以：
   * 7 = 4+2+1 (rwx)
   * 5 = 4+0+1 (r-x, 也就是只读和执行，没有写入权限)
   * 6 = 4+2+0 (rw-)

  所以 `775` 的含义是：
  你和你的组可以随便折腾，但“路人甲”只能看、能运行，但不能改你的代码。这在生产环境中很常见，比如
  Web 服务器的网页文件。

  2. 企业级权限工具：ACL (Access Control List)
  你猜对了，简单的 chmod 确实不够用。比如：一个文件需要给 A 组读写权限，给 B
  组只读权限，还要给特定的程序员 C 开启执行权限。
   * 命令：getfacl (查看) 和 setfacl (设置)。
   * 示例：setfacl -m u:klee:rwx file.txt (精准给 klee 用户开小灶)。

   3. 逻辑判断 (`if...then...else`): 让脚本有“大脑”。

  1. 查找内存占用前五的进程
  当你发现系统卡顿（内存不足）时：
   * 命令：ps aux --sort=-%mem | head -n 6
   * 拆解：
       * ps aux: 列出系统所有运行中的进程。
       * --sort=-%mem: 按内存百分比（%mem）降序（-号）排序。
       * head -n 6: 只显示前 6 行（第 1 行是标题，下面 5 行是前五名）。
   * 场景：你可以一眼抓到那个“吃内存”的怪兽（比如 Chrome 或 Java 程序）。


  2. 查找特定端口的程序（比如 80 端口）
   * 命令：ss -tunlp | grep :80
   * 拆解：
       * ss: socket statistics，这是现在最推荐的查网络连接的工具。
       * -t: TCP 连接。
       * -u: UDP 连接。
       * -n: 以数字形式显示端口（不把 80 显示成 http）。
       * -l: 仅显示监听中的端口。
       * -p: 显示进程信息（Process），这个最关键，它能告诉你谁在用这个端口。
   * 场景：当你启动 Nginx 发现报错“端口被占用”时，用这个命令就能发现是谁在背后搞鬼。

  我来为你逐一拆解：

  ---


  1. CPU 和 GPU 的排序显示
   * CPU: ps aux --sort=-%cpu 是完全正确的。
   * GPU: 这是一个“陷阱”。ps 命令只能看到 CPU 和 内存 的使用情况，它无法直接看到 GPU。
       * 为什么？ 因为 ps 读取的是 /proc 文件系统中的进程信息，而 GPU 属于外部硬件加速器。
       * Arch 用户方案：如果你用的是 NVIDIA 显卡，你应该用 nvidia-smi。如果你想实时排序看谁占显存：
          nvidia-smi --query-compute-apps=pid,process_name,used_memory --format=csv
       * 通用方案：安装 nvtop（Arch 官方库里有：sudo pacman -S nvtop），它就像 GPU 版的 htop。

  ---


  2. 如果缺少减号 (-) 会发生什么？
   * 有减号 (`-%)：代表 降序 (Descending)。从大到小排列（最耗资源的在最上面）。
   * 没减号 (%)：代表 升序 (Ascending)。从小到大排列（最闲置的在最上面）。
   * 结果：如果你不加减号，你的屏幕会被成百上千个“占用 0.0%
     CPU”的系统守护进程刷屏，而真正卡顿的进程被排到了最底端，你一眼看不见。

  ---


  3. 如果没有管道符 (|) 和后面的东西会发生什么？
  如果你只运行 ps aux --sort=-%cpu：
   * 现象：终端会瞬间吐出 几百行 文本（你系统里所有的进程）。
   * 后果：由于屏幕滚动太快，你必须手动往回翻才能看到最顶端（最耗资源）的进程。
   * 管道符的意义：管道 | 就像一个漏斗。head -n 6
     的作用是：“我不管你后面排了多少人，我只要看领头的这 6 个。” 这样你的终端清清爽爽，只显示重点。

  ---


  4. 关于日志保存 (>>)
  你的写法：ps aux --sort=-%mem | head -n 999 >> ~/logs.txt
   * 逻辑正确：>> 代表 追加 (Append)。它不会删除之前的内容，而是把新数据接到旧数据后面。
   * 进阶细节：
       * 如果你用 > (单箭头)：它会 覆盖 (Overwrite) 整个文件。每次运行，旧的日志就没了。
       * DevOps 建议：在存日志时，最好加上时间戳，否则你不知道这条记录是哪天的：
          echo "--- Report at $(date) ---" >> ~/logs.txt && ps aux --sort=-%mem | head -n 10 >>
  ~/logs.txt

  1. 查询一个程序占用了哪些端口？
  你有两种最专业的“工程师级”打法：

  方法 A：使用 `ss` (Socket Statistics) —— 现代、极快


   1 # 假设你想查名为 "samba" 或 "smbd" 的程序
   2 ss -tunlp | grep smbd
   * 参数拆解：
       * -t: 查看 TCP 端口。
       * -u: 查看 UDP 端口（Samba 经常用 UDP 137/138）。
       * -n: 以数字显示端口号（不翻译成文字，速度更快）。
       * -l: 只看正在监听（Listening）的端口。
       * -p: 显示进程（Process）的名字和 PID。这是找到“谁”占了端口的关键。


  方法 B：使用 `lsof` (List Open Files) —— 万能、精细
  在 Linux 中，“万物皆文件”，网络连接也是一个文件描述符。


   1 # 查找特定程序名的网络占用
   2 sudo lsof -i -P -n | grep smbd
   * 参数拆解：
       * -i: 筛选互联网套接字（网络连接）。
       * -P: 强制显示端口号（不把 80 显示成 http）。
       * -n: 强制显示 IP（不解析域名）。

  ---

  2. 启用 Samba 后，端口会自动向局域网开放吗？
  答案是：“服务已在听，大门未必开。”

  在 DevOps 中，一个服务要能被外面访问，必须满足两个条件：


  条件 1：监听地址 (Binding Address)
  服务必须监听在 0.0.0.0（所有网卡）或者你的局域网 IP 上，而不是 127.0.0.1（仅本机）。
   * 检查方法：运行 ss -tunlp。
       * 如果看到 127.0.0.1:445：只有你自己能用。
       * 如果看到 0.0.0.0:445 或 *:445：它正张开双臂迎接全网。


  条件 2：防火墙 (Firewall) —— 这是 Arch 用户的必修课
  即使 Samba 已经在 0.0.0.0 监听，如果你的系统防火墙没开门，局域网里的其他电脑依然连不上。
   * 如果你没装防火墙：默认是开放的。
   * 如果你装了 `ufw`：
      sudo ufw allow samba
   * 如果你装了 `firewalld`：
      sudo firewall-cmd --add-service=samba --permanent
      sudo firewall-cmd --reload

✦ 这就是 Shell 脚本中的“特殊字符陷阱”！在 DevOps 领域，这种“懒”法经常会导致各种意外报错。

  作为 Arch + Fish 用户，你刚才的操作触发了以下三个 Shell 的核心禁忌：

  1. 为什么会报错？（拆解你的错误）


   1. 没有引号的换行：
      Shell（无论是 Fish 还是 Bash）默认以 回车（换行符）
  作为一条命令的终点。当你回车输入第二行时，Fish 认为第一条 echo 已经结束了。它会接着把第二行开头的
  2 当成一个新的命令去执行，所以它报错说 未知命令：2。


   2. 特殊符号 `<NAME>`：
      在所有类 Unix Shell 中，< 代表 输入重定向（从文件读取内容）。Fish 看到 <NAME>
  会去你的文件夹里找一个叫 NAME 的文件。如果找不到，它就会发火。


   3. 反引号 `` ` `` (Backticks)：
      反引号在 Shell 里代表 命令替换。它会尝试先运行里面的 ss -tunlp | grep
  <NAME>，并把结果塞进你的 echo 里。但显然，你现在只是想把这段话当成普通文字存起来。

  ---

  🛠️ 怎么做才是正确的？（Fish Shell 懒人法）


  在 Fish 中，最稳妥、最“懒”且最专业的做法是使用 单引号 `'...'`，因为它会告诉
  Shell：“不要动里面的任何东西，它们只是普通文字。”

  方法 A：使用 `printf` (最推荐，DevOps 标配)
  printf 比 echo 更强大，处理换行更专业。


   1 printf '### Network Troubleshooting (Advanced)
   2 - `ss -tunlp | grep <NAME>` : 查找程序占用的端口 (Modern)
   3 - `lsof -i -P -n` : 列出所有网络连接
   4 - `firewall-cmd --list-all` : 查看防火墙开放了哪些服务
   5 - `curl -I <IP:PORT>` : 仅请求 HTTP 头部，检查服务是否存活' >> linux-commands-cheatsheet.md


  方法 B：使用双引号 (如果你需要里面包含单引号)
  如果你用了双引号，里面的 < 和 > 依然可能被解析，所以对于包含复杂符号的文档，单引号是王道。

  📘 第一部分：核心教学 —— Shell 脚本的“三大支柱”


  在 DevOps
  自动化中，脚本不能是“写死”的。它需要根据不同的环境（变量）、不同的需求（输入）做出反应。


  1. 变量 (Variables)：脚本的记忆
   * 定义：NAME="Klee"（注意：等号两边绝对不能有空格！这是 Bash/Zsh/Fish 的通病）。
   * 引用：echo $NAME 或 echo "${NAME}_server"（加花括号是为了防止变量名和后面的字符连在一起）。


  2. 用户输入 (read)：脚本的耳朵
   * 用法：read -p "请输入你想查询的端口号: " PORT
   * 效果：脚本会停下来等你输入，然后把你输入的内容存进变量 PORT。


  3. 算术运算 (Arithmetic)：脚本的头脑
   * 写法：SUM=$((1 + 1))（注意是双圆括号 (( ))）。
   * 进阶：在脚本里计算 CPU 占用百分比或磁盘剩余空间时，这非常有用。
