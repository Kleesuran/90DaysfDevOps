  📘 第一部分：核心概念教学

  在开始写计划之前，我们先搞清楚两个最基础的词：DevOps 和 Cloud Engineering。

  1. 什么是 DevOps？
  很多人以为 DevOps 就是 Jenkins + Docker + K8s，其实它更像是一种文化和流程的“翻译官”。
   * Dev (Development)：想要快速发布新功能。
   * Ops (Operations)：想要系统稳如老狗，不挂掉。
   * DevOps：打破两者的隔阂，通过自动化（Automation）让开发和运维同步，实现“高质量的快速交付”。

  2. 什么是 Cloud Engineering（云工程）？
  如果说 DevOps 是流程，那 Cloud 就是战场。
  过去我们要买物理服务器、拉网线（On-Premise）；现在我们通过代码直接在 AWS/Azure
  上“变出”成千上万台服务器。这就是 Infrastructure as Code (IaC，基础设施即代码)。

  📘 第一部分：核心教学 —— Linux 文件系统与存储自动化

  在 Day 01 中，你提到过修改 fstab 来挂载 Windows
  游戏硬盘。那是手工时代的巅峰，但现在我们要进入自动化时代。

  1. 什么是 `fstab`？
  fstab (File Systems Table) 是 Linux 内核启动时读取的配置文件。它决定了哪些分区挂载到哪个目录下。
   * 痛点：如果你手动改错了 fstab，Arch 可能会进入紧急模式（Emergency Mode），让你进不去系统。
   * DevOps 方案：使用 UUID（Universally Unique Identifier）而不是 /dev/sdb1 这种会变动的名称。

  2. 新工具教导：`lsblk` 与 `blkid`
   * `lsblk`: 以树状列出所有块设备（硬盘、分区）。
       * 用法：lsblk -f (列出文件系统类型和 UUID)。
   * `blkid`: 查看更详细的块设备属性，通常用于获取精准的 UUID。

  📘 第一部分：纠偏与进阶 —— 权限的“数学逻辑”

  1. 权限数值纠偏 (非常重要！)
  在 Linux 中，权限是基于二进制的：
   * Read (r) = 2^2 = 4
   * Write (w) = 2^1 = 2
   * Execute (x) = 2^0 = 1

  所以：
   * 7 = 4+2+1 (rwx)
   * 5 = 4+0+1 (r-x, 也就是只读和执行，没有写入权限)
   * 6 = 4+2+0 (rw-)

  所以 `775` 的含义是：
  你和你的组可以随便折腾，但“路人甲”只能看、能运行，但不能改你的代码。这在生产环境中很常见，比如
  Web 服务器的网页文件。

  2. 企业级权限工具：ACL (Access Control List)
  你猜对了，简单的 chmod 确实不够用。比如：一个文件需要给 A 组读写权限，给 B
  组只读权限，还要给特定的程序员 C 开启执行权限。
   * 命令：getfacl (查看) 和 setfacl (设置)。
   * 示例：setfacl -m u:klee:rwx file.txt (精准给 klee 用户开小灶)。

   3. 逻辑判断 (`if...then...else`): 让脚本有“大脑”。

  1. 查找内存占用前五的进程
  当你发现系统卡顿（内存不足）时：
   * 命令：ps aux --sort=-%mem | head -n 6
   * 拆解：
       * ps aux: 列出系统所有运行中的进程。
       * --sort=-%mem: 按内存百分比（%mem）降序（-号）排序。
       * head -n 6: 只显示前 6 行（第 1 行是标题，下面 5 行是前五名）。
   * 场景：你可以一眼抓到那个“吃内存”的怪兽（比如 Chrome 或 Java 程序）。


  2. 查找特定端口的程序（比如 80 端口）
   * 命令：ss -tunlp | grep :80
   * 拆解：
       * ss: socket statistics，这是现在最推荐的查网络连接的工具。
       * -t: TCP 连接。
       * -u: UDP 连接。
       * -n: 以数字形式显示端口（不把 80 显示成 http）。
       * -l: 仅显示监听中的端口。
       * -p: 显示进程信息（Process），这个最关键，它能告诉你谁在用这个端口。
   * 场景：当你启动 Nginx 发现报错“端口被占用”时，用这个命令就能发现是谁在背后搞鬼。

  我来为你逐一拆解：

  ---


  1. CPU 和 GPU 的排序显示
   * CPU: ps aux --sort=-%cpu 是完全正确的。
   * GPU: 这是一个“陷阱”。ps 命令只能看到 CPU 和 内存 的使用情况，它无法直接看到 GPU。
       * 为什么？ 因为 ps 读取的是 /proc 文件系统中的进程信息，而 GPU 属于外部硬件加速器。
       * Arch 用户方案：如果你用的是 NVIDIA 显卡，你应该用 nvidia-smi。如果你想实时排序看谁占显存：
          nvidia-smi --query-compute-apps=pid,process_name,used_memory --format=csv
       * 通用方案：安装 nvtop（Arch 官方库里有：sudo pacman -S nvtop），它就像 GPU 版的 htop。

  ---


  2. 如果缺少减号 (-) 会发生什么？
   * 有减号 (`-%)：代表 降序 (Descending)。从大到小排列（最耗资源的在最上面）。
   * 没减号 (%)：代表 升序 (Ascending)。从小到大排列（最闲置的在最上面）。
   * 结果：如果你不加减号，你的屏幕会被成百上千个“占用 0.0%
     CPU”的系统守护进程刷屏，而真正卡顿的进程被排到了最底端，你一眼看不见。

  ---


  3. 如果没有管道符 (|) 和后面的东西会发生什么？
  如果你只运行 ps aux --sort=-%cpu：
   * 现象：终端会瞬间吐出 几百行 文本（你系统里所有的进程）。
   * 后果：由于屏幕滚动太快，你必须手动往回翻才能看到最顶端（最耗资源）的进程。
   * 管道符的意义：管道 | 就像一个漏斗。head -n 6
     的作用是：“我不管你后面排了多少人，我只要看领头的这 6 个。” 这样你的终端清清爽爽，只显示重点。

  ---


  4. 关于日志保存 (>>)
  你的写法：ps aux --sort=-%mem | head -n 999 >> ~/logs.txt
   * 逻辑正确：>> 代表 追加 (Append)。它不会删除之前的内容，而是把新数据接到旧数据后面。
   * 进阶细节：
       * 如果你用 > (单箭头)：它会 覆盖 (Overwrite) 整个文件。每次运行，旧的日志就没了。
       * DevOps 建议：在存日志时，最好加上时间戳，否则你不知道这条记录是哪天的：
          echo "--- Report at $(date) ---" >> ~/logs.txt && ps aux --sort=-%mem | head -n 10 >>
  ~/logs.txt

  1. 查询一个程序占用了哪些端口？
  你有两种最专业的“工程师级”打法：

  方法 A：使用 `ss` (Socket Statistics) —— 现代、极快


   1 # 假设你想查名为 "samba" 或 "smbd" 的程序
   2 ss -tunlp | grep smbd
   * 参数拆解：
       * -t: 查看 TCP 端口。
       * -u: 查看 UDP 端口（Samba 经常用 UDP 137/138）。
       * -n: 以数字显示端口号（不翻译成文字，速度更快）。
       * -l: 只看正在监听（Listening）的端口。
       * -p: 显示进程（Process）的名字和 PID。这是找到“谁”占了端口的关键。


  方法 B：使用 `lsof` (List Open Files) —— 万能、精细
  在 Linux 中，“万物皆文件”，网络连接也是一个文件描述符。


   1 # 查找特定程序名的网络占用
   2 sudo lsof -i -P -n | grep smbd
   * 参数拆解：
       * -i: 筛选互联网套接字（网络连接）。
       * -P: 强制显示端口号（不把 80 显示成 http）。
       * -n: 强制显示 IP（不解析域名）。

  ---

  2. 启用 Samba 后，端口会自动向局域网开放吗？
  答案是：“服务已在听，大门未必开。”

  在 DevOps 中，一个服务要能被外面访问，必须满足两个条件：


  条件 1：监听地址 (Binding Address)
  服务必须监听在 0.0.0.0（所有网卡）或者你的局域网 IP 上，而不是 127.0.0.1（仅本机）。
   * 检查方法：运行 ss -tunlp。
       * 如果看到 127.0.0.1:445：只有你自己能用。
       * 如果看到 0.0.0.0:445 或 *:445：它正张开双臂迎接全网。


  条件 2：防火墙 (Firewall) —— 这是 Arch 用户的必修课
  即使 Samba 已经在 0.0.0.0 监听，如果你的系统防火墙没开门，局域网里的其他电脑依然连不上。
   * 如果你没装防火墙：默认是开放的。
   * 如果你装了 `ufw`：
      sudo ufw allow samba
   * 如果你装了 `firewalld`：
      sudo firewall-cmd --add-service=samba --permanent
      sudo firewall-cmd --reload

  作为 Arch + Fish 用户，你刚才的操作触发了以下三个 Shell 的核心禁忌：

  1. 为什么会报错？（拆解你的错误）


   1. 没有引号的换行：
      Shell（无论是 Fish 还是 Bash）默认以 回车（换行符）
  作为一条命令的终点。当你回车输入第二行时，Fish 认为第一条 echo 已经结束了。它会接着把第二行开头的
  2 当成一个新的命令去执行，所以它报错说 未知命令：2。


   2. 特殊符号 `<NAME>`：
      在所有类 Unix Shell 中，< 代表 输入重定向（从文件读取内容）。Fish 看到 <NAME>
  会去你的文件夹里找一个叫 NAME 的文件。如果找不到，它就会发火。


   3. 反引号 `` ` `` (Backticks)：
      反引号在 Shell 里代表 命令替换。它会尝试先运行里面的 ss -tunlp | grep
  <NAME>，并把结果塞进你的 echo 里。但显然，你现在只是想把这段话当成普通文字存起来。

  ---

  🛠️ 怎么做才是正确的？（Fish Shell 懒人法）


  在 Fish 中，最稳妥、最“懒”且最专业的做法是使用 单引号 `'...'`，因为它会告诉
  Shell：“不要动里面的任何东西，它们只是普通文字。”

  方法 A：使用 `printf` (最推荐，DevOps 标配)
  printf 比 echo 更强大，处理换行更专业。


   1 printf '### Network Troubleshooting (Advanced)
   2 - `ss -tunlp | grep <NAME>` : 查找程序占用的端口 (Modern)
   3 - `lsof -i -P -n` : 列出所有网络连接

   5 - `curl -I <IP:PORT>` : 仅请求 HTTP 头部，检查服务是否存活' >> linux-commands-cheatsheet.md


  方法 B：使用双引号 (如果你需要里面包含单引号)
  如果你用了双引号，里面的 < 和 > 依然可能被解析，所以对于包含复杂符号的文档，单引号是王道。

  📘 第一部分：核心教学 —— Shell 脚本的“三大支柱”


  在 DevOps
  自动化中，脚本不能是“写死”的。它需要根据不同的环境（变量）、不同的需求（输入）做出反应。


  1. 变量 (Variables)：脚本的记忆
   * 定义：NAME="Klee"（注意：等号两边绝对不能有空格！这是 Bash/Zsh/Fish 的通病）。
   * 引用：echo $NAME 或 echo "${NAME}_server"（加花括号是为了防止变量名和后面的字符连在一起）。


  2. 用户输入 (read)：脚本的耳朵
   * 用法：read -p "请输入你想查询的端口号: " PORT
   * 效果：脚本会停下来等你输入，然后把你输入的内容存进变量 PORT。


  3. 算术运算 (Arithmetic)：脚本的头脑
   * 写法：SUM=$((1 + 1))（注意是双圆括号 (( ))）。
   * 进阶：在脚本里计算 CPU 占用百分比或磁盘剩余空间时，这非常有用。

  📘 核心概念：本地 (Local) vs 远程 (Remote)

   1. git commit (本地)：就像你在本地写日记，写完锁在抽屉里。只有你自己能看。
   2. git push (远程)：就像你把日记复印了一份，寄到了 GitHub 的服务器上。这时候全世界（或者你的团队）才能在网页上看到。

  🛠️ 动手操作：把代码推送到 GitHub

  作为 Arch 用户，我们直接用命令行解决。请按顺序执行：

  第一步：确认你的远程仓库（Remote）
  看看你的电脑知不知道 GitHub 仓库的地址：
   1 git remote -v
   * 预期输出：你应该能看到以 origin 开头的两行，后面跟着你的 GitHub 仓库 URL。
   * 如果没有输出：说明你还没关联远程仓库（如果你是 git clone 下来的，通常已经自动关联好了）。

  📘 1. 关于 Fork 和冲突 (Conflicts)

   * 你会和原作者冲突吗？ 不会。 因为你现在是在推送代码到 你自己的 Fork 仓库（也就是 Kleesuran/90DaysfDevOps）。这就像你在自己的笔记本上记笔记，原作者看不见，也管不着。

  📘 2. 拆解 git push origin master

  这句命令看起来重复，其实逻辑非常严密：

  git push [目的地] [哪个分支]

   1. origin (目的地)：
      这是一个“别名”。它代表了你刚才设置的长串 SSH 地址 (git@github.com...)。在 Git 里，默认的远程仓库统称为 origin（起源）。
   2. master (分支名)：
      这代表你要把你电脑上的 master 分支推送上去。

  为什么要写两次？
  其实完整的命令是 git push origin master:master（把本地 master 推送到远程 master）。因为名字一样，所以我们简写成了 git push origin master。
   * DevOps 场景：如果你想把你本地的代码推送到远程一个叫 test 的分支，你就会写 git push origin master:test。

  ---

  📘 3. Day 04 脚本的“翻车现场” (Age = abc)

  你的直觉很准，但 Shell 的表现可能比你想象的更“奇葩”：

  在 Bash/Fish 的算术运算 $((65 - AGE)) 中：
   * 如果 AGE="abc"，Shell 默认会把 非数字字符串看作 0。
   * 结果：它不会报错退出，而是会一本正经地告诉你：“你还有 65 年退休！”（65 - 0 = 65）。
   * DevOps 教训：这比报错更危险！因为它给出了一个错误的正确答案。在生产环境中，这可能导致自动扩容脚本把服务器关掉。

   18 echo "正在统计 $DIR 的占用情况..."
   19 du -sh "$DIR"
  (写完后按 Esc，输入 :wq 保存退出)

  📘 刚才发生了什么？（知识点梳理）

   1. read -p：这是脚本的“耳朵”。它会在屏幕上打印一段话，然后等着你输入，并把输入的内容存进后面的变量名里。
   2. $(( ... ))：这是脚本的“计算器”。在 Bash 里，直接写 65 - AGE 它会当成文字，必须套在双括号里才算数学题。
   3. 引号的学问：
       * 双引号 "$DIR"：如果你输入的文件夹路径带空格，双引号能保证它不被打断。（职业习惯：变量最好都加双引号）。

  ---

  在 Linux 中，新写的脚本只是个“文本文件”，我们要给它“执行权限”：

   2 chmod +x user_input.sh
   3
   4 # 运行它！
   5 ./user_input.sh

  📘 第一部分：核心教学 —— if 语句的“魔法公式”

  在 Bash 脚本中，逻辑判断的语法看起来有点怪（有很多方括号），但它的逻辑非常严密：

  1. 基本结构
   1 if [ 条件 ]; then
   2     # 如果条件成立，执行这里的命令
   3 else
   4     # 如果条件不成立，执行这里的命令
   5 fi
   * ⚠️ 职业提醒：方括号 [ 和 ] 的两边必须有空格！
       * if [$A == $B] ❌ (报错)
       * if [ $A == $B ] ✅ (正确)

  2. 常用的“判官”参数
  作为 DevOps 工程师，你最常判断的是文件和数值：
   * -f 文件名: 判断这个文件是否存在。
   * -d 目录名: 判断这个目录是否存在。
   * -eq: 等于 (Equal)
   * -ne: 不等于 (Not Equal)
   * -gt: 大于 (Greater Than)

  📝 第二部分：第 5 日任务目标 (2026 版)

  我查看了 2026/day-05/README.md，今天的任务是编写一个 “智能环境检查脚本”。

  任务要求：
   1. 创建一个脚本 env_checker.sh。
   2. 判断文件：检查 /etc/passwd 是否存在（这个文件存储了系统用户信息）。如果存在，输出“系统用户文件正常”。
   3. 判断数值：让用户输入一个数字。如果大于 100，输出“这个数字太大了！”；否则输出“数字在安全范围内”。
   4. DevOps 进阶：检查你的电脑是否安装了 docker。如果安装了，输出版本号；如果没有，提示“请安装 Docker 以继续”。

   18 fi

  ❓ 第四部分：今日挑战提问

  让我们来拆解这两个关于 “逻辑大脑” 的真相：

  ---

  📘 1. 逻辑连接符：猜对了一半！

  在 Bash 中，如果你想同时检查两个条件，有两种写法：

   * 旧式写法 ([ ... ])：使用 -a (And) 和 -o (Or)。
       * if [ $NUM -gt 10 -a $NUM -lt 20 ]
   * 现代写法 ([[ ... ]])：这是 DevOps 工程师的首选！ 它支持你猜的 && 和 ||，而且更安全。
       * if [[ $NUM -gt 10 && $NUM -lt 20 ]]

  💡 职业习惯： 永远优先使用 双中括号 [[ ... ]]，因为它不会因为变量为空而轻易崩溃。

  ---

  📘 2. 为什么会报错“需要一元运算符”？

  这个报错是初学者的噩梦，但原理其实极其简单：

  当你直接按回车，变量 $NUM 就是空的。
  Bash 在执行时会进行“文字替换”：
   1. 你的代码：if [ $NUM -gt 100 ]
   2. 替换后：if [  -gt 100 ] （中间那个 $NUM 消失了！）

  这时候，[ 命令（它其实是一个程序）发现 -gt 前面空空如也，它不知道拿什么去跟 100 比，于是就愤怒地报错：“需要一元运算符（即左边的数字）！”

  至于为什么输出了“安全区间”？
  因为 if 语句运行报错了，返回了“失败”。既然失败了，它就自动跳进了 else 分支。由于 $NUM 是空的，打印出来就变成了 (%)。

  ---

  🛠️ 怎么修复？（Day 05 的“防弹”写法）

  为了让你的脚本更健壮（Robust），我们有两种防御手段：

  方案 A：给变量加双引号 (简单有效)
   1 if [ "$NUM" -gt 100 ]; then
   * 这样即使 $NUM 是空的，Bash 也会把它看作 ""（空字符串），虽然比大小还是会报错，但不会崩得那么难看。

  方案 B：使用 [[ ... ]] (最推荐)
   1 if [[ $NUM -gt 100 ]]; then
   * 双中括号会自动处理变量为空的情况，不会报刚才那个低级错误。

  ---

  📘 1. 小括号 () 在 Bash 里的真正作用：沙盒 (Subshell)

  在 Bash 中，单括号 () 的首要作用是创建一个 子 Shell (Subshell)。

   * 功能：在一个完全独立的“进程沙盒”里运行命令。
   * 现象：在 () 里面修改的变量、切换的目录，都不会影响外面的主脚本。
   * 示例：
   1     NAME="Klee"
   2     (
   3         NAME="Arch"
   4         echo "括号内: $NAME" # 输出 Arch
   5     )
   6     echo "括号外: $NAME" # 输出 Klee (没被改变！)

  ---

  📘 2. 为什么 Bash 不用 () 做比较？

  在 C、Java 或 Python 里，if (a > b) 是天经地义的。但 Bash 为什么不行？

  历史原因：[ 曾经是一个真正的程序
  在 Linux 的远古时代，[ 并不是 Bash 的语法，而是一个位于 /usr/bin/[ 的独立二进制文件（现在你依然可以在 Arch 里运行 ls -l /usr/bin/[ 看到它）。

   * if [ $A -eq $B ] 本质上是：“运行 [ 这个程序，并把 $A, -eq, $B, ] 当作参数传给它。”
   * 因为 ( 已经被用来表示“子 Shell”了，如果再拿它来做比较，Shell 就会彻底搞混：它不知道你是想开个沙盒，还是想比大小。

  ---

  📘 3. 既然如此，有没有像 C 语言那样的括号用法？

  有！那就是双小括号 (( ))。

  如果你只进行纯数字的比较，双小括号就是你的“救星”，它不需要 -gt 这种奇怪的字母，可以直接用 > 和 <。

   * Bash 的“数学实验室”：
   1     if (( $NUM > 100 )); then
   2         echo "这看起来终于像 C 语言了！"
   3     fi
   * 区别：
       * [[ ... ]]：处理字符串、文件、正则表达式的多面手。
       * (( ... ))：处理纯数学计算的专家。

  ---

  📘 4. 总结：括号家族的大合影

  为了让你不再搞混，我为你总结了这套“括号暗号”：


  ┌───────┬──────────┬─────────────────────────────┬──────────────────┐
  │ 符号  │ 称呼     │ 用途                        │ 例子             │
  ├───────┼──────────┼─────────────────────────────┼──────────────────┤
  │ ()    │ 单小括号 │ 子 Shell (沙盒运行)         │ (cd /tmp && ls)  │
  │ $( )  │ 命令替换 │ 拿结果 (把命令输出存进变量) │ FILES=$(ls)      │
  │ (( )) │ 双小括号 │ 纯算术 (比大小、加减乘除)   │ (( a = 1 + 1 ))  │
  │ [ ]   │ 单中括号 │ 老式判断 (不推荐，容易崩)   │ [ $a -eq 1 ]     │
  │ [[ ]] │ 双中括号 │ 现代判断 (DevOps 推荐)      │ [[ $a == "ok" ]] │
  └───────┴──────────┴─────────────────────────────┴──────────────────┘

✦ 🔍 找到症结了！作为“初学者”，你犯了几个非常经典且具有代表性的错误。

  我们要把你的脚本从“一个能跑的文本”变成“一个合格的生产工具”，需要修正以下 4 个关键点：

  ---

  1. 致命伤：命令替换 $( ... ) 的误用
   * 你的代码：$(fastfetch) 和 $(ps aux ...)
   * 问题：在 Bash 中，$( ) 的意思是：“执行这个命令，并把它的输出当作一条新命令去运行”。
       * 比如：如果 fastfetch 输出的第一行是 OS: Arch Linux，Bash 就会尝试运行一个叫 OS: 的命令。由于没有 OS: 这个命令，它就会报错。
   * 修正：直接写命令名即可，不要加 $( )。
       * ✅ fastfetch
       * ✅ ps aux --sort=-%mem | head -n 5

  ---

  2. 语法细节：方括号的空格
   * 你的代码：[[ ... -gt 80]]
   * 问题：80 和 ]] 之间必须有一个空格！Bash 的解析器非常固执，它分不清 80]] 是什么意思。
   * 修正：[[ ... -gt 80 ]]

  ---

  3. read 命令的变量引用
   * 你的代码：read $answer
   * 问题：read 是要把输入存进变量里，这里不应该加 $。加了 $ 代表你想读取“变量 $answer 的内容”作为存储目标的变量名。
   * 修正：read answer

  ---

  4. 路径引用 (Shebang)
   * 你的代码：#!/usr/bin/bash
   * 提示：虽然在 Arch 上这没问题，但在更通用的 Linux 环境（如 Ubuntu）中，最标准的写法是 #!/bin/bash。

  📘 第一部分：核心教学 —— 文件操作的“三板斧”

  在 Linux 中，处理文本文件不一定要打开 nvim，命令行才是最快的工具。

  1. 重定向的艺术 (> vs >>)
   * > (覆盖): 像“推土机”，先把文件清空，再写入新内容。
   * >> (追加): 像“接龙”，把内容接在文件末尾。

  2. 阅读器的选择 (cat, head, tail)
   * cat: 一口气打印全文件（适合小文件）。
   * head -n 5: 只看前 5 行（适合看配置文件头部）。
   * tail -n 5: 只看最后 5 行（DevOps 调试日志的命根子）。

  3. 瑞士军刀：tee
   * 痛点：有时候你想把输出存进文件，但同时也想在屏幕上看到它。
   * 方案：echo "Hello" | tee -a notes.txt
       * -a 代表 append (追加)，如果不加，它也会像 > 一样覆盖。
